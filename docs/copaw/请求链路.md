从请求到达 console（前端/接口）开始 到 模型输出返回并落盘 的完整执行线路串起来（包含关键分支：命令、重连、后台任务）。

## 0）入口：你在 console 输入并发送
通常 Web Console 会对后端发起：

- POST /api/console/chat （SSE 流式返回）
  对应后端入口是 console.py:68-151 。

（同一套系统也支持标准 AgentApp 的 POST /api/agent/process ，但 Web Console 这条链路走的是 /api/console/chat 。）

## 1）后端启动与上下文：CLI → FastAPI → MultiAgentManager
你运行 CLI：

- copaw app → uvicorn 启动 FastAPI： app_cmd.py:54-93
  FastAPI app 初始化（lifespan）时会：

- 创建 MultiAgentManager ，并按配置预启动 agent workspaces： _app.py:148-239
  请求进入后，中间件会注入 agent_id（用于多 agent 路由）：

- AgentContextMiddleware 从 /api/agents/{agentId}/... 或 X-Agent-Id 取 agent_id，并写入 contextvar： agent_scoped.py:15-50
## 2）/api/console/chat：解析请求 → 找到 workspace → 找到 console channel
入口函数 post_console_chat 主要做这些事：

1. 定位 workspace（也就是目标 agent 的运行时实例）

   - workspace = await get_agent_for_request(request) ： console.py:82
   - 逻辑优先级：显式参数 > request.state.agent_id > X-Agent-Id header > config.active_agent： agent_context.py:22-83

2. 获取 console channel
   - console_channel = await workspace.channel_manager.get_channel("console") ： console.py:83-88
3. 把前端请求转换成 native_payload（channel 友好格式）
   - _extract_session_and_payload(...) ： console.py:32-65
4. 确定 session_id，并创建/获取 chat 记录（用于会话持久化与 run_key）
   - session_id = console_channel.resolve_session_id(...) ： console.py:93-96
   - chat = await workspace.chat_manager.get_or_create_chat(...) ： console.py:104-109
     这里 chat.id （ChatSpec.id）会作为后续 TaskTracker 的 run_key ，用来支持断线重连与多订阅者。

## 3）TaskTracker：把一次 chat 运行变成“可重连的后台流”
/api/console/chat 并不是直接在请求协程里跑完整推理，而是交给 TaskTracker：

- 非重连： tracker.attach_or_start(chat.id, native_payload, console_channel.stream_one) ： console.py:124-128
- 重连： tracker.attach(chat.id) ： console.py:116-123
  TaskTracker 的核心机制：

- attach_or_start 创建一个后台 _producer task，循环消费 console_channel.stream_one(payload) 产生的 SSE 字符串，并广播给所有订阅 queue，同时写入 buffer（用于重连回放）： task_tracker.py:142-208
- HTTP 响应则是 StreamingResponse(event_generator()) ，从 tracker.stream_from_queue() 持续 yield：
    - console.py:130-151
    - task_tracker.py:210-230
## 4）ConsoleChannel.stream_one：native_payload → AgentRequest → 调 runner.stream_query → SSE
后台 producer 实际调用的是：

- ConsoleChannel.stream_one(payload) ： console/channel.py:272-361
  关键步骤：

1. 把上传引用（图片/文件等）解析为本地路径，并做 no-text debounce（例如先上传图再发文字时合并）
   - _resolve_console_upload_refs ： console/channel.py:199-247
   - _apply_no_text_debounce 在 BaseChannel 里： base.py:247-279
2. 构建 AgentRequest （runtime 标准结构）
   - build_agent_request_from_native ： console/channel.py:249-270
3. 调用 self._process(request) 获取 Event 流，并逐条转成 SSE 行
   - async for event in self._process(request): ... yield f"data: {data}\n\n" ： console/channel.py:309-330
     这里的 self._process 是 ChannelManager 注入的 ProcessHandler ，对 workspace 来说就是：

   - process = make_process_from_runner(runner) → 直接返回 runner.stream_query ： channels/utils.py:121-133
     也就是说： console chat 的核心“跑模型”入口就是 workspace.runner.stream_query 。

另外 ConsoleChannel 还会在 message completed 事件时把内容打印到服务端 stdout（给“控制台频道”看的），并可能推送到前端的 push store：

- 打印与 push 的触发点： console/channel.py:331-347 、 console/channel.py:459-493

## 5）Runner 层：stream_query（外部库）→ AgentRunner.query_handler（项目内）
runner.stream_query 本体来自 agentscope_runtime.engine.runner.Runner （外部依赖，不在仓库里）。但它会委托到你项目里覆写的：

- AgentRunner.query_handler(...) ： runner.py:188-403
  query_handler 的关键流程：

1. 取出 query / session_id / user_id / channel 等 ： runner.py:197-203
2. 工具审批（tool guard）优先处理
   - _resolve_pending_approval ： runner.py:95-187
   - 若审批被拒，会清理 session memory 并直接 yield 一个拒绝 Msg： runner.py:204-218
3. 命令通道（/compact、/clear 等）优先分流
   - if query and _is_command(query): run_command_path(...) ： runner.py:219-224
4. 正常对话：构造 CoPawAgent 并执行
   - agent = CoPawAgent(... memory_manager=self.memory_manager ...) ： runner.py:277-299
   - 载入 session state（历史消息、压缩摘要等）： runner.py:341-353
   - rebuild sys prompt（确保 prompt 取最新磁盘文件）： runner.py:355-359
   - stream_printing_messages(agents=[agent], coroutine_task=agent(msgs)) 产生 Msg 流并 yield 出去： runner.py:360-365
     这些 Msg 最终会被 Runner.stream_query （外部库）转换成你在 ConsoleChannel 里看到的 Event（包含 message/response/tool 等对象），再变成 SSE。

5. finally：保存 session state、更新 chat
   - save_session_state(...) ： runner.py:392-399
   - chat_manager.update_chat(chat) ： runner.py:400-401
## 6）Agent 内部：MemoryManager/工具暴露/自动压缩（影响一次请求的中后段）
当 CoPawAgent 构造完成后（它继承/扩展了 ReActAgent），会在初始化里做 MemoryManager 接线：

- _setup_memory_manager ：注入 memory、回写 model/formatter、注册 memory_search 工具： react_agent.py:308-341
- _register_hooks ：注册 pre_reasoning 的自动压缩 hook： react_agent.py:361-371
- 自动压缩 hook 里会触发 compact_memory / check_context 等： memory_compaction.py:62-210
  这部分决定了： 一次请求中是否会发生“上下文满了→自动压缩→写入 compressed_summary→继续推理/回复” 。

## 一句话总结整条链路（最短可读版）
Web Console 输入 → POST /api/console/chat → get_agent_for_request 拿 Workspace → TaskTracker 启后台 producer → ConsoleChannel.stream_one 把输入转成 AgentRequest → 调 workspace.runner.stream_query → AgentRunner.query_handler 构造并运行 CoPawAgent （含 MemoryManager/工具/hook）→ 产出 Msg → 外部 Runner 包装成 Event → ConsoleChannel 转 SSE → TaskTracker 广播给 HTTP SSE → 前端实时显示，同时服务端 stdout 也会打印完成消息。
